记录一下LLM提示词工程框架

ICIO、CRISPE、BROKE、RASCEF 四个提示词框架

发布时间: 2026-01-25
最后更新: 2026-03-14
阅读时长: 12 分钟

记录一下 LLM 提示词工程框架

我平时更常用的方式其实是“口语化提问 + 追问迭代”。但这种方式有个问题：同一个问题，模型有时能给到我想要的答案，有时就开始跑题、堆概念，甚至自己编设定。很多时候不是模型不会，是我没把「要交付什么」说清楚。

这篇笔记就把几个我觉得常见、也确实用得上的提示词框架记一下：ICIO、BROKE、CRISPE、RASCEF。我更倾向把它们当成一组检查项：当输出总是答不到点上，通常是少写了某个关键要素。

我常用的偷懒结论（先记这几句）

  • 只要把交付写清楚：用 ICIO
  • 需要目标、交付物、还能迭代收敛：用 BROKE
  • 需要强约束、稳定格式：用 改良版 CRISPE
  • 需要按步骤执行、最怕漏步骤/跳步骤：用 RASCEF

1) ICIO：把需求写清楚的最小集合

ICIO 基本就是“把需求写清楚”的最小集合，特别适合翻译、摘要、信息提取这种标准任务。它解决的往往不是“模型不会”，而是“我没说清楚它要怎么交付”——尤其是输出格式。

我自己也会拿它当自检：输出不稳定时，先看是不是少了 Output Indicator（比如我想要表格/清单/分点结论，但我只说了“总结一下”）。

核心要素

  • Instruction (指令): 明确告诉 AI 需要执行的具体操作。
  • Context (背景): 提供任务的背景信息，帮助 AI 理解语境，减少歧义。
  • Input Data (输入数据): 明确需要处理的具体内容。
  • Output Indicator (输出指引): 规定输出的格式和限制。

适用场景

  • 文本处理: 翻译、润色、摘要、改写。
  • 信息提取: 从长文中提取关键数据。
  • 格式转换: 将文本转换为表格、JSON 等。

案例：生成会议纪要

Instruction: 请总结以下会议记录，提炼出核心观点和待办事项。
Context: 这是一次关于公司年度营销策略的内部讨论会议，参会人员包括市场部全体成员。
Input Data:
[此处粘贴会议录音转文字稿…]
Output Indicator: 请用 Markdown 列表输出，分为“核心观点”和“Action Items”两部分，尽量短句，不要复述细节。

我偷懒时就用一句话概括：“基于【背景】，把【输入】按【格式】做【动作】。”

2) BROKE：把问题当项目做（目标、交付、迭代）

BROKE 更像“按项目思路把问题写清楚”，强调目标、交付物和可落地性，适合商业分析、策略方案这种需要对齐口径的场景。

它最有用的一点是 Evolve：我可以把第一次回答当草稿，然后继续追问它“怎么更能落地”。我更常用的方式是先让它给一个可用框架，再沿着 KR 和 Evolve 把细节补全。

核心要素

  • Background (背景): 阐述任务的起因、环境及现状。
  • Role (角色): 设定 AI 的身份。
  • Objectives (目标): 明确希望达成的目标。
  • Key Results (关键结果): 定义成功的标准或具体的交付物。
  • Evolve (演进/改进): 允许在对话中根据 AI 的回答进行迭代和优化，或者要求 AI 自我反思并给出改进建议。

适用场景

  • 商业分析: 市场调研、竞品分析。
  • 策略制定: 运营计划、危机公关方案。
  • 问题解决: 针对具体痛点寻求解决方案。

案例：制定客户留存方案

Background: 我们的 SaaS 产品在用户注册后的第一个月内流失率高达 40%，主要反馈是“上手太难”。
Role: 请扮演一位专注于用户增长（Growth Hacker）的产品运营专家。
Objectives: 我们需要制定一个针对新用户的“30天引导计划”，旨在将首月流失率降低至20%以内。
Key Results: 请产出一份包含5个具体触达节点（邮件或应用内弹窗）的执行方案，说明每个节点的内容和预期效果。
Evolve: 在给出方案后，请列出该方案可能面临的3个最大执行障碍，并给出预案。

我自己对 BROKE 的印象是：如果方案“看起来很完整，但执行细节不够”，BROKE 往往能把交付物和迭代点补齐。

偷懒写法：“背景是【…】；目标是【…】；交付物要长这样【…】；先给方案，再自我挑 3 个最大风险并补预案。”

3) CRISPE：同名不同配方（我分两种用）

CRISPE 很常见，但不同文章里经常定义不一致。我这里按两种主流用法拆开记：经典版（偏创意发散） 和 改良版（偏工程精准）。我不纠结哪个更“正宗”，只关心什么时候好用。

版本 A：经典版 CRISPE（偏创意与发散）

这个版本的关键是先给角色、给语气，再让它多试几次。它的目标是“多给候选”，不是“一次到位”。

核心要素：

  • Capacity and Role (能力与角色): 定义 AI 应扮演的角色。
  • Insight (洞察/背景): 提供详细的背景信息和上下文。
  • Statement (陈述/指令): 明确具体的任务指令。
  • Personality (个性): 设定 AI 的语气和风格（例如：幽默、严谨）。
  • Experiment (实验/尝试): 要求 AI 提供多个不同的选项或方案（例如：“给我三个不同的标题”）。

适用场景：创意写作、头脑风暴、市场营销文案。

版本 B：改良版 CRISPE（偏精准与工程）

改良版对我更实用：少强调语气设定，多讲“边界”。它把不稳定的部分（个性/发散）弱化掉，换成两件更可靠的东西：限制条件（Parameters） 和 示例（Examples）。我想要固定格式输出，或者需要严格按既定口径来，通常就会用这个版本。

偷懒写法：“角色设为【…】。在【上下文】里完成【任务】。输出风格是【…】。限制是【…】。参考示例【…】。”

核心要素：

  • Capacity and Role (能力与角色): 定义 AI 的身份。
  • Relevant Context (相关背景): 提供任务背景。
  • Instruction (指令): 核心任务指令。
  • Style (风格): 定义输出的风格或结构（例如：正式报告、JSON格式）。
  • Parameters (参数/限制): 设定具体的约束条件（例如：字数限制、语言、必选词汇）。
  • Examples (示例): 提供 Few-shot examples（少样本提示）来引导 AI 输出。

适用场景：代码生成、数据格式化、技术文档编写。

4) RASCEF：改良版 CRISPE + 明确步骤（更工程化）

RASCEF 我会理解成“改良版 CRISPE + 明确步骤”。它最关键的是 Steps：先把任务拆成一串明确动作，再让模型照着走。

这套对编程、排障、流程设计这类任务很有效，因为这些任务最怕的不是写不出来，而是漏步骤、跳步骤、偷换前提。把 Steps 写清楚，能明显减少“看起来写了很多，但关键一步没做”。

核心要素

  • Role (角色): 指定 AI 的角色。
  • Action (动作): 明确具体的动作。
  • Steps (步骤): 列出完成任务所需的具体步骤。很多复杂任务能不能做对，就卡在这里。
  • Context (背景): 提供必要的上下文。
  • Examples (示例): 提供示例（Few-shot prompting）来引导 AI 的输出风格或格式。
  • Format (格式): 指定最终输出的格式。

演变关系：从改良版 CRISPE 到 RASCEF

  • 继承: RASCEF 继承了改良版 CRISPE 的核心逻辑（Role, Context, Examples）。
  • 强化: 把 Instruction 拆成 Action + Steps，先把“要做哪些动作、按什么顺序做”说明白。
  • 结论: 当结果需要更可控、更接近“照着操作就能跑起来”，RASCEF 往往比一句大而全的指令更靠谱。

适用场景

  • 编程开发: 代码生成、Code Review。
  • 技术文档: 编写操作手册、API 文档。
  • 复杂流程执行: 需要严格按顺序执行的任务。

案例：编写 Python 数据处理脚本

Role: 请扮演一位资深 Python 数据工程师。
Action: 编写一个 Python 脚本，用于清洗 CSV 数据。
Context: 数据文件名为 data.csv，包含 ‘date’, ‘sales’, ‘region’ 三列。部分 ‘sales’ 数据缺失或格式错误。
Steps:

  1. 读取 CSV 文件。
  2. 将 ‘date’ 列转换为 datetime 对象。
  3. 填充 ‘sales’ 列的缺失值为 0，并剔除负数。
  4. 按 ‘region’ 分组统计总销售额。
  5. 将结果保存为 summary.csv。

偷懒写法：“角色设为【…】。要做【动作】。按这几步做【步骤 1/2/3】。输入背景是【…】。输出格式必须是【…】。”

5) 框架 vs 口语化：我一般怎么选

现实里更常见的确实是用对话式的方式去问，而不是每次都把 ICIO/RASCEF 写满。我也一样。框架的意义不是格式本身，而是把那些容易被默认掉的条件补全：受众是谁、要什么形式、哪些不能写、怎么才算做完。

什么时候我会用框架（或半框架）

下面这些场景，结构化输入的收益通常非常高：

  • 高风险/高成本：对外发布、对客户交付、写代码要直接进仓库、写合同/公告等，错一次代价大。
  • 复杂任务：需要拆步骤、需要多轮迭代、需要同时满足多约束（受众、格式、口径、合规）。
  • 强格式输出：要表格/JSON/接口文档/测试用例/清单，或者要脚本、命令、配置这种“拿过去就能跑”的东西。
  • 长上下文：资料很多（会议纪要、PRD、日志、论文），不结构化很容易丢重点。
  • 可复用/可交接：同一类任务会反复做，或需要团队复用；框架就是可复制的“工作流”。
  • 需要可衡量结果：比如 BROKE 的 KR，明确“什么叫成功”，能减少来回扯皮。

我自己常用的“半框架”最低标准是把三件事说清楚：要做什么（任务）/ 基于什么（关键上下文）/ 怎么才算好（输出要求与约束）。很多时候到这一步就够稳了。

什么时候口语化更划算

口语化对话更适合这些情况：

  • 探索与发散：先找方向、要灵感、要候选方案（“给我 10 个思路我再挑”）。
  • 低成本小任务：简单解释、快速润色、列要点、起标题、翻译几句话。
  • 需求还不确定：还在“想清楚要什么”的阶段，框架反而会限制探索。
  • 可以接受迭代：先要个草稿，再通过追问把它收敛到可用版本。

不写框架也能稳一点：口语化的最小结构

我发现口语化问题里最容易缺的几类信息，补齐之后输出会稳定很多：

  • 目标：最终用途是什么（发邮件、写汇报、做决策、写代码、做 PPT）。
  • 受众与语气：给谁看、希望什么风格（面试官/老板/客户/同事；正式/口语/简洁）。
  • 输入材料：已有的事实、数据、原文、约束条件（哪怕只有几条也很有用）。
  • 输出边界：字数、结构、是否给多方案、是否要引用来源、是否要可执行步骤。
  • 缺信息时的策略：明确约定“如果信息不足，先提出最多 3 个问题再继续”。

本质上，这些要素就是框架的核心，只是换成更自然的表达方式。

我常用的几种提问套路（比较口语，但更稳定）

  1. 先要“草稿”，再要“改稿”
    第一句只求快速覆盖面，第二句再加约束收敛质量。比如：
  • “帮我写一份产品周报，先给一个结构和要点。”
  • “把语气改得更专业一些，控制在 300 字以内，突出本周结果和下周风险。”
  1. 用“反例/禁区”减少跑偏
  • “不要写空话套话（如‘持续优化’、‘赋能’），尽量用可量化的描述。”
  • “不要引入我没给的事实，如果需要数据请先问我。”
  1. 把难点点出来，让模型重点处理
  • “我最拿不准的是 A 和 B 的取舍，请对比它们的利弊，并给出推荐与理由。”
  1. 要求输出可验证的东西
  • “给我一份检查清单，让我能自查这份内容是否达标。”
  • “列出这些判断依赖的前提条件/假设。”

几句常用问法（我会直接复制改一改）

  • 通用问答：
    “目标是【目标】。现有信息是【要点 1/2/3】。请按【输出形式】输出。如果信息不够，请先提出最多 3 个问题再继续。”
  • 写作/汇报：
    “请撰写【文种】，给【对象】看，语气【风格】，控制在【字数】。必须包含【要点】，避免【禁区】。先给一个版本，再给 3 条修改建议。”
  • 方案/决策：
    “背景是【背景】，需要做【决策】。约束是【预算/时间/资源】。请给 3 个方案：每个方案写清步骤、成本、收益、风险、适用条件，最后给出推荐与理由。”
  • 编程/工程问题：
    “环境是【环境/语言/框架】，遇到的问题是【问题】。报错/日志如下：【粘贴】。期望结果是【期望】。请先按排查顺序列可能原因，再给一个最小可行的修复方案。”

6) 用 AI 辅助生成 prompt（我更常把它当“提示词编辑器”）

这一节是我日常最常用的小技巧：把 AI 当“提示词整理器”。很多时候任务本身不难，难的是需求描述太散，关键约束没说全。先让模型把口语化需求整理成结构化 prompt，再拿这个 prompt 去跑任务，稳定性通常会好很多。

我比较推荐分两步：先写 prompt，再执行任务。第一步只做整理（不输出答案），第二步再按整理后的 prompt 去生成结果。这样能避免它一边组织提示词一边提前开始回答，导致结构跑偏。

一个顺手的流程

  1. 先用口语把需求说清楚（不用很完整）。
  2. 让模型反问几个关键信息（一般 3 个就够）。
  3. 让它按某个框架输出“最终 prompt”（只输出提示词，不要输出答案）。
  4. 把最终 prompt 原封不动贴回去，让模型开始执行任务。

让它先当“提示词编辑”

例如我会这样写：

请先不要回答任务内容，先把需求整理成一个提示词。
按【ICIO / 改良版 CRISPE / RASCEF】格式输出。
如果信息不够，先提出最多 3 个问题。
输出只要提示词本身，不要附加解释。

一般来说：

  • 文字处理、摘要、提取信息：用 ICIO 就够了。
  • 要求固定格式、要遵守口径：用改良版 CRISPE。
  • 需要按步骤做、需要可执行结果：用 RASCEF。

“把我这段话变成 prompt”的例子

比如我随口写：

我想写一封邮件给老板说下这个月做了啥，别太啰嗦，重点写结果。

让它输出成 ICIO 风格的提示词，大概会长这样（示意）：

Instruction: 写一封月度工作总结邮件
Context: 收件人是直属老板，语气专业但不生硬
Input Data: 本月完成的事情如下（后续补充列表）
Output Indicator: 200-300 字，先写结果，再写关键过程，最后写下月计划和风险点，避免空话

再把“本月完成的事情”补进去，就能直接用了。

加边界：别让它乱编

如果我经常遇到“写得很顺，但事实不对”，会把下面两句塞进提示词里：

  • “不要补充未提供的事实或数据，缺的部分先提问确认。”
  • “如有任何假设，请单独列出来并等待确认。”

反向用法：拿一段好答案，反推 prompt

有时候我手上已经有一段“风格很符合预期”的文本（比如同事的周报、以前写过的文案），也可以把它当示例，让模型反推一个可复用的提示词：

这是一个我满意的输出样例（贴上去）。
请反推一个提示词，让模型以后也能写出类似风格。
要求：包含风格约束、结构约束、禁区，最后给一段能直接拿去问的版本。

这个方法对“写作风格”“口径”“输出结构”特别好用。

总结：四个框架怎么选（我自己的对照表）

框架 核心特点 适用场景 记忆口诀
ICIO 简单直接，要素少 翻译、摘要、简单任务 指令+背景+输入+输出
BROKE 目标导向，注重迭代 商业策略、解决问题 背景+角色+目标+结果+演进
CRISPE 经典版发散，改良版精准 创意写作 vs 工程任务 角色+洞察/背景+指令+个性/风格
RASCEF 更工程化，强调步骤 编程、复杂流程、高精度任务 角色+动作+步骤+背景+示例+格式

我的经验是：不用为了“写提示词”而写提示词。大多数日常问题，口语化问法 + 一两轮追问就够了；但当结果有明确要求（要可执行、要可复用、要对齐口径），框架能显著减少来回试错。最常用的组合还是 ICIO（把交付说清） + 改良版 CRISPE / RASCEF（把边界和步骤说清）。
